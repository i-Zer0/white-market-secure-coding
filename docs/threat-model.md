# White Market 위협 모델

## 1. 범위와 보호 자산

보호 대상은 사용자 계정과 전화번호, 세션 및 CSRF 토큰, 상품·채팅·거래 데이터, 관리자 권한과 감사 로그, 업로드 이미지, 암호화 DB 백업이다. 주요 신뢰 경계는 브라우저-HTTP 서버, 일반 사용자-관리자 기능, 앱-SQLite, 앱-SMS·지오코딩 공급자, 운영 서버-백업 저장소이다.

## 3. 거래 흐름의 권한 규칙

구매자만 판매 중 상품에 거래를 요청한다. 판매자가 수락하면 해당 거래와 상품이 예약 중으로 바뀌고, 다른 미처리 요청은 거절된다. 구매자만 예약 거래를 완료할 수 있다. 후기와 별점은 채팅 이력이 있거나 완료 거래의 당사자인 경우에만 허용된다. 가격 제안과 거래 약속은 해당 상품의 구매자·판매자 관계를 서버에서 다시 검증하며 생성·응답·변경 이력을 별도 테이블에 기록한다.

근거 위치: `app.py:3860-4021`, `app.py:4036-4052`, `app.py:4065-4186`, `app.py:4287-4304`.

## 4. 남은 위험과 운영 통제

- SMS Webhook의 인증 방식은 공급자별로 다르므로 운영 연결 시 API 키를 비밀 저장소에 보관하고 공급자 서명·IP 제한을 추가해야 한다.
- 인메모리 Rate Limit은 단일 프로세스 기준이다. 여러 서버 인스턴스를 운영할 때는 Redis 같은 중앙 저장소가 필요하다.
- Caddy 자동 인증서는 실제 도메인 DNS와 80/443 포트가 연결되어야 발급된다.
- 암호화는 백업 파일을 보호하지만 실행 중인 SQLite와 서버 메모리까지 암호화하지 않는다. 운영 서버의 디스크 암호화와 최소 권한 계정을 함께 사용한다.
- 현재 교육용 HTTP 서버는 단일 인스턴스 배포를 전제로 한다. 실제 서비스에서는 관측성, 중앙 로그, 비밀 관리자, 복구 훈련을 추가한다.
